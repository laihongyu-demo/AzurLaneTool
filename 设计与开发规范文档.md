# DataInsight 设计与开发规范文档

## 1. 项目概述与技术栈

本项目旨在开发一个轻量级桌面应用程序，用于**展示 SQLite 数据库中的数据**，并集成**部分 Python 后端计算能力**（如统计汇总、数据变换、指标分析）。应用采用分层架构，确保代码清晰、可维护且易于扩展。

**技术栈**：
- **Python** (>=3.8) —— 主编程语言
- **Qt5** (PyQt5) —— GUI 框架，用于构建用户界面
- **SQLite** —— 嵌入式数据库，存储应用数据
- （可选）`matplotlib` 或 `PyQtChart` —— 数据可视化图表库

## 2. 架构设计原则与分层规范

### 2.1 目的
规范开发流程，保证代码一致性、可维护性，并严格遵循分层架构原则，降低各模块间的耦合度。

### 2.2 适用范围
本规范涵盖以下全部层级：
- **视图层 (View)**：基于 Qt Widgets 的 UI 组件
- **业务逻辑层 (Business/Service)**：核心计算、数据处理、流程编排
- **数据访问层 (Repository/DAO)**：封装 SQLite 的 CRUD 操作
- **模型层 (Model)**：定义数据实体类（DTO/POJO）
- **工具层 (Utility)**：通用辅助函数、常量、异常定义

### 2.3 核心原则

| 原则 | 说明 |
|------|------|
| **分层职责清晰** | 视图层只负责 UI 展示与用户交互，业务层处理计算与逻辑，数据层专用于数据库访问，禁止跨层职责模糊。 |
| **跨层交互可控** | 上层可以调用下层（View → Service → Repository），严禁反向依赖（如下层导入上层模块）。同一层级内部可相互调用，但需避免循环依赖。 |
| **代码可复用** | 通用功能（如日期格式化、ID 生成器）必须抽取至工具层，避免重复实现。 |
| **逻辑可测试** | 业务逻辑层与数据访问层应设计为无 UI 依赖的普通 Python 类，便于编写单元测试。 |

## 3. 功能模块详细设计

### 3.1 数据展示模块

**前端实现**：
- 主窗口继承 `QMainWindow`，使用 `QLabel` 显示文本/数值指标。
- 数据图表可采用：
  - `PyQt5.QtChart` 原生图表（性能较好）
  - 嵌入 `matplotlib` 的 `FigureCanvasQTAgg`（功能更丰富）

**数据流向**（遵循分层调用）：
```
用户点击刷新按钮（View）
    ↓ 触发信号
业务逻辑层（Service）的 refresh_data() 方法
    ↓ 调用
数据访问层（Repository）的 query_raw_data() 方法
    ↓ 执行 SQL 查询
SQLite 数据库返回原始数据（如 List[Dict] 或 List[Model]）
    ↓ 回到业务层
业务层进行聚合计算、统计、格式化
    ↓ 返回处理后的 DTO 或直接可展示的数据结构
视图层接收结果并更新 QLabel 文本 / 刷新图表
```

**代码示例（视图层信号连接）**：
```python
# views/main_window.py
class MainWindow(QMainWindow):
    def __init__(self, data_service):
        super().__init__()
        self.data_service = data_service
        self.refresh_btn.clicked.connect(self.on_refresh_clicked)

    def on_refresh_clicked(self):
        summary = self.data_service.get_summary_stats()
        self.label_total.setText(f"总计: {summary.total}")
        self.chart.update_data(summary.trend_data)
```

### 3.2 后端计算与数据操作模块

#### 3.2.1 计算功能（位于业务逻辑层）
所有从数据库查询后的二次处理、计算、分析必须封装在 `services/` 下的类中。

**示例**：
```python
# services/calc_service.py
class CalcService:
    def compute_statistics(self, records: List[ShipModel]) -> Dict:
        values = [r.capacity for r in records]
        return {
            "mean": sum(values) / len(values),
            "max": max(values),
            "min": min(values)
        }
```

#### 3.2.2 数据持久化功能（位于数据访问层）
所有 SQLite 的 CRUD 操作必须封装在 `repositories/` 下。每个 Repository 对应一个主要实体。

**禁止**在视图层或业务层出现任何 SQL 字符串。

**示例**：
```python
# repositories/ship_repository.py
import sqlite3
from typing import List, Optional
from models.ship_model import ShipModel

class ShipRepository:
    def __init__(self, db_path: str):
        self.db_path = db_path

    def find_all(self) -> List[ShipModel]:
        with sqlite3.connect(self.db_path) as conn:
            conn.row_factory = sqlite3.Row
            cursor = conn.execute("SELECT id, name, capacity, launch_date FROM ships")
            return [ShipModel(**dict(row)) for row in cursor.fetchall()]

    def insert(self, ship: ShipModel) -> int:
        with sqlite3.connect(self.db_path) as conn:
            cursor = conn.execute(
                "INSERT INTO ships (name, capacity, launch_date) VALUES (?, ?, ?)",
                (ship.name, ship.capacity, ship.launch_date)
            )
            return cursor.lastrowid
```

#### 3.2.3 大型 SQL 语句管理

对于长度较长、逻辑复杂且**无需动态修改参数**（即静态 SQL）的查询语句，应遵循以下规范：

- **存放位置**：所有 `.sql` 文件统一存放在项目根目录下的 `sql/` 文件夹中。
- **文件命名**：使用小写字母和下划线，描述语句用途，如 `monthly_report.sql`、`export_inactive_users.sql`。
- **语句特征**：此类 SQL 通常不含占位符（`?` 或 `%s`），执行时直接使用。如需参数化（如日期范围），推荐仍使用参数化查询，不应通过字符串拼接。
- **加载方式**：在数据访问层中，通过工具函数读取 `.sql` 文件内容，然后执行。禁止在 Repository 中硬编码大段 SQL。

**工具层辅助函数示例**（`utils/sql_loader.py`）：
```python
import os

def load_sql_file(filename: str) -> str:
    """
    从 sql/ 目录加载 SQL 文件内容。

    Args:
        filename: 文件名（如 'monthly_report.sql'）

    Returns:
        SQL 文件中的完整字符串。

    Raises:
        FileNotFoundError: 当文件不存在时抛出。
    """
    base_dir = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
    sql_path = os.path.join(base_dir, 'sql', filename)
    with open(sql_path, 'r', encoding='utf-8') as f:
        return f.read()
```

**Repository 中使用示例**：
```python
# repositories/ship_repository.py
from utils.sql_loader import load_sql_file

class ShipRepository:
    # ... 其他方法 ...

    def generate_monthly_report(self) -> List[Dict]:
        """执行月报统计查询（SQL 从外部文件加载）"""
        sql = load_sql_file("monthly_report.sql")
        with sqlite3.connect(self.db_path) as conn:
            conn.row_factory = sqlite3.Row
            cursor = conn.execute(sql)
            return [dict(row) for row in cursor.fetchall()]
```

- **注意事项**：
  - 即使 SQL 是静态的，也仍需确保查询结果不会因外部输入而引发 SQL 注入（因为无用户参数拼接，风险较低，但应避免动态组装）。
  - 若后续某个静态 SQL 需要参数化，应重构为带占位符的版本，并将参数通过 `execute(sql, params)` 传入，同时将该 SQL 文件视为“模板”，可考虑使用 `sqlparse` 或简单字符串格式化（谨慎使用）。

## 4. 编码与风格规范

### 4.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | `PascalCase` | `ShipRepository`, `MainWindow`, `DataService` |
| 方法/函数名 | `camelCase` | `calculateAverage`, `fetchUserData`, `onRefreshClicked` |
| 变量/属性名 | `snake_case` | `total_count`, `connection_string`, `is_visible` |
| 常量名 | `UPPER_CASE_WITH_UNDERSCORES` | `MAX_CONNECTIONS`, `DEFAULT_QUERY_TIMEOUT` |
| 私有成员（类内部） | 前缀单下划线 `_` | `_internal_method`, `_cache_dict` |

### 4.2 文档与注释

- **文档字符串 (Docstring)**：所有公开的类、方法、函数必须编写 docstring，遵循 [PEP 257](https://peps.python.org/pep-0257/) 风格。推荐使用三引号 `"""`，明确说明功能、参数、返回值和可能抛出的异常。

```python
def fetch_by_id(self, record_id: int) -> Optional[ShipModel]:
    """
    根据ID查询单条船舶记录。

    Args:
        record_id: 船舶记录的唯一标识ID。

    Returns:
        如果存在则返回 ShipModel 对象，否则返回 None。

    Raises:
        ValueError: 当 record_id <= 0 时抛出。
    """
    if record_id <= 0:
        raise ValueError("record_id must be positive")
    # 实现...
```

- **行内注释**：复杂算法或关键业务逻辑处需添加简明扼要的注释，解释 **“为什么这么做”**，而非描述代码本身。

```python
# 使用二分查找优化，因为列表已排序且长度超过1000
index = bisect_left(sorted_prices, target)
```

### 4.3 代码格式

- 严格遵守 **[PEP 8](https://peps.python.org/pep-0008/)**。
- 缩进：**4个空格**，禁止使用 Tab。
- 每行代码长度尽量不超过 **120** 个字符（文档字符串、长URL等可适当放宽至 120~150）。
- 导入语句必须分组排序，组间空一行，每组内按模块名称字母顺序排列。

```python
# 1. 标准库导入
import os
import sys
from typing import List, Dict, Optional

# 2. 第三方库导入
from PyQt5 import QtWidgets, QtCore
import numpy as np

# 3. 项目内部模块导入
from models.ship_model import ShipModel
from repositories.ship_repository import ShipRepository
from utils.constants import MAX_RETRY_COUNT
from utils.helpers import format_currency
```

### 4.4 异常处理

- 在数据访问层捕获 `sqlite3.Error` 并转换为自定义业务异常（如 `DatabaseError`），避免将底层数据库异常直接抛至视图层。
- 视图层可捕获业务异常并通过 `QMessageBox` 向用户展示友好错误信息。

```python
# utils/exceptions.py
class DatabaseError(Exception):
    """数据库操作相关异常"""
    pass

# repositories/ship_repository.py
def find_all(self):
    try:
        # sqlite操作
    except sqlite3.Error as e:
        raise DatabaseError(f"查询失败: {e}")
```

### 4.5 资源管理

- 数据库连接：推荐使用上下文管理器（`with sqlite3.connect(...)`）自动提交/回滚，避免手动关闭。
- Qt 对象：充分利用 Qt 的父子对象机制管理内存；对于长时间运行的线程或定时器，需在析构时显式停止。

## 5. 版本控制与 Git 提交规范

本项目采用 **GitHub** 托管，代码量预估在 1 万行以内，为单机个人使用的轻量级应用。尽管如此，为保持提交历史的清晰、便于回溯和协作（如未来可能增加贡献者），所有开发人员（含个人）必须遵循以下版本控制流程。

### 5.1 分支策略

采用简化的 **Git Flow**，适合小型项目：

- **`main`** 分支：生产就绪代码，始终处于可运行状态。**严禁直接在 main 分支上进行任何代码编写、修改或提交操作**，确保 main 分支始终保持可部署的稳定状态。所有对 main 分支的变更只能通过从 `develop` 分支发起的 Pull Request 合并，且需经过代码审核。
- **`develop`** 分支：集成分支，包含最新开发功能。**所有日常开发工作必须在 develop 分支上进行**，或基于 develop 创建特性分支后进行。develop 分支是开发活动的核心基线，任何功能变更最终都必须汇入 develop。
- **`feature/xxx`** 分支：用于单个功能或 bug 修复。从 `develop` 切出，完成后合并回 `develop`。**功能分支仅存在于本地环境，禁止直接 push 到远程仓库**（详见 5.2 节）。

> ⚠️ **重要提示**：本项目不采用"个人开发者可省略 develop 分支"的简化模式。无论团队规模大小，均须严格遵守上述分支策略，确保 main 分支的稳定性与代码变更的可追溯性。

**推荐操作流程**：
```bash
# 克隆仓库
git clone https://github.com/yourname/DataInsight.git

# 切换到 develop 分支
git checkout develop
git pull origin develop

# 基于 develop 创建特性分支
git checkout -b feature/add-chart-widget

# 在特性分支上开发并提交
git add .
git commit -m "feat(view): 添加数据刷新按钮及图表联动"

# 开发完成后，本地合并到 develop（确保基于最新 develop）
git checkout develop
git pull origin develop
git merge --no-ff feature/add-chart-widget

# 本地验证合并结果
python main.py  # 确保应用可正常运行

# 推送本地 develop 到远程（详见 5.2 节审核流程）
git push origin develop

# 删除已合并的本地特性分支
git branch -d feature/add-chart-widget
```

### 5.2 功能分支与代码审核规范

#### 5.2.1 功能分支 Push 限制

所有以 `feature/` 为前缀命名的功能分支，**禁止直接 push 到远程仓库**。功能分支仅存在于开发者本地环境，用于隔离开发中的代码变更，避免远程仓库出现未经验证的功能分支。

**禁止操作示例**：
```bash
# ❌ 禁止：直接推送功能分支到远程
git push origin feature/add-chart-widget
```

#### 5.2.2 代码合并与审核流程

功能分支开发完成后，必须遵循以下流程将代码变更合并至远程 `develop` 分支：

1. **本地合并**：开发者先在本地将功能分支合并到 `develop` 分支，确保代码无冲突且功能正常。
2. **本地验证**：在本地 `develop` 分支上运行应用，确认合并后功能正常、无回归问题。
3. **发起 Pull Request**：在 GitHub 上向远程 `develop` 分支发起 Pull Request（PR），PR 标题与描述需清晰写明变更内容、影响范围及测试情况。
4. **人工审核**：由指定的代码审核人员对 PR 进行审核，重点检查代码质量、规范遵循情况及潜在问题。
5. **审核通过后合并**：审核通过后，方可将 PR 合并至远程 `develop` 分支。审核未通过则需根据反馈修改后重新提交。
6. **清理分支**：合并完成后，删除本地的功能分支。

**完整操作流程**：
```bash
# 1. 从 develop 创建特性分支
git checkout develop
git pull origin develop
git checkout -b feature/add-chart-widget

# 2. 在特性分支上开发并提交
git add .
git commit -m "feat(view): 添加数据刷新按钮及图表联动"

# 3. 开发完成后，本地合并到 develop
git checkout develop
git pull origin develop
git merge --no-ff feature/add-chart-widget

# 4. 本地验证合并结果
python main.py  # 确保应用可正常运行

# 5. 推送本地 develop 到远程
git push origin develop

# 6. 在 GitHub 上向 develop 发起 Pull Request
#    （通过 GitHub Web 界面操作，填写变更说明）

# 7. 代码审核通过后合并 PR

# 8. 删除已合并的本地特性分支
git branch -d feature/add-chart-widget
```

#### 5.2.3 Pull Request 规范

- **标题格式**：与 Commit Message 保持一致，使用 `<type>(<scope>): <subject>` 格式。
- **描述内容**：需包含变更说明、影响范围、本地测试情况及关联的 issue（如有）。
- **审核要求**：至少一名代码审核人员确认后方可合并。对于个人项目，建议通过 PR 自我审查以养成良好习惯。
- **审核重点**：
  - 代码是否符合分层架构规范（无跨层调用、无视图层 SQL 等）
  - 代码风格是否符合 PEP 8 及本规范命名要求
  - 公开类/方法是否已添加 docstring
  - 是否存在潜在的异常未处理或资源泄漏问题

#### 5.2.4 Push 与 Pull 规范

| 操作 | 规范 | 说明 |
|------|------|------|
| **Push** | 禁止直接 push `feature/*` 分支到远程 | 功能分支仅存于本地，合并至 develop 后通过 PR 提交 |
| **Push** | 禁止直接 push 到 `main` 分支 | main 分支的所有变更必须通过从 develop 发起的 PR 合并 |
| **Push** | 推送 `develop` 前须先 `git pull` | 避免覆盖他人提交，推送前始终执行 `git pull origin develop` |
| **Pull** | 开发前先 `git pull origin develop` | 确保本地代码基于最新远程 develop，减少合并冲突 |
| **Pull** | PR 审核期间及时拉取更新 | 若 PR 存在冲突，需本地解决后重新推送 |

### 5.3 Commit Message 规范

提交信息必须遵循 **Conventional Commits** 轻量化格式，便于自动生成 changelog 和快速定位变更类型。

**格式**：
```
<type>(<scope>): <subject>

[可选正文]

[可选脚注]
```

- **type**（必选）：说明提交类别，使用以下词汇之一：
  - `feat`：新功能
  - `fix`：修复 bug
  - `docs`：仅文档变更（如 README、本规范文档）
  - `style`：代码格式调整（空格、缩进、分号等，不影响逻辑）
  - `refactor`：重构（既不是新功能也不是修复 bug 的代码变动）
  - `perf`：性能优化
  - `test`：添加或修正测试
  - `chore`：构建过程、辅助工具、依赖更新等（不涉及源代码或测试）

- **scope**（可选）：影响范围，如 `view`、`repository`、`sql`、`utils`。对于全局性改动可使用 `*`。

- **subject**（必选）：简短描述（不超过 72 字符），使用中文或英文，保持一致性。**不以句号结尾**，使用动词开头（如“添加”、“修复”、“更新”）。

**示例**：
```
feat(view): 添加数据刷新按钮及图表联动

fix(repository): 修复 SQLite 连接未正确关闭导致的锁问题

docs(readme): 更新安装说明与依赖列表

refactor(service): 将计算逻辑从视图层迁移至业务层

chore(deps): 升级 PyQt5 至 5.15.9
```

- [ ] 不提交敏感信息（如数据库密码、个人令牌）。对于 SQLite 数据库文件（`.db` 或 `.sqlite`）应加入 `.gitignore`。

### 5.6 GitHub 协作流程

虽然是个人项目，但若未来邀请协作者，遵循以下流程：

1. **Fork** 主仓库到个人账号。
2. 在个人 fork 上基于 `develop` 创建 `feature/xxx` 分支进行开发。
3. 开发完成后，在本地将功能分支合并到 `develop`，验证功能正常。
4. 推送本地 `develop` 到个人 fork，向主仓库的 `develop` 发起 **Pull Request**。
5. PR 标题与描述中写明变更内容，并关联相关 issue（如有）。
6. 由指定代码审核人员进行审核，审核通过后方可合并至主仓库的 `develop` 分支。

> **注意**：功能分支（`feature/*`）禁止直接 push 到远程仓库，必须通过本地合并至 develop 后以 PR 方式提交审核（详见 5.2 节）。
